# Community Guidelines

FOSSHack Jabalpur 2026 is a community-driven event. These guidelines help ensure a productive, respectful, and collaborative experience for all participants, mentors, and organizers.

These guidelines complement the [Code of Conduct](CODE_OF_CONDUCT.md) and apply to all community interactions throughout the event.

---

## Communication Standards

### Be Clear and Constructive

- Write clearly and professionally in all communications
- Provide context when asking questions or reporting issues
- When giving feedback, focus on the work, not the person
- Acknowledge and appreciate contributions from others

### Use Appropriate Channels

- Use GitHub Issues for project-specific questions, bug reports, and feature suggestions
- Use GitHub Pull Requests for code and documentation contributions
- Use designated event communication channels (Discord, mailing lists) for general discussion
- Do not use public channels to share sensitive or private information

### Stay On Topic

- Keep discussions relevant to the event, projects, or open-source development
- Avoid off-topic conversations that distract from the event's purpose
- Do not use event channels for promotional or commercial purposes

---

## Collaboration Principles

### Open and Transparent Development

- Work openly — commit code to your public repository regularly
- Share progress, challenges, and decisions with the community
- Document your reasoning in commit messages, pull requests, and issue comments

### Respect Others' Work

- Do not copy or reuse code without proper attribution
- Follow the license terms of any open-source libraries or assets you use
- Credit contributors appropriately in your project README and commit history

### Support Fellow Participants

- Help others when you have the knowledge to do so
- Share resources, tutorials, and tools that may benefit the community
- Do not gatekeep knowledge or create unnecessary barriers for newcomers

---

## Repository and Code Standards

### Repository Hygiene

- Keep your repository organized and well-documented
- Use meaningful branch names and commit messages
- Do not commit sensitive information such as passwords, API keys, or tokens
- Use `.gitignore` to exclude build artifacts and local configuration files

### Issue and Pull Request Etiquette

- Search for existing issues before opening a new one
- Provide sufficient detail in issue descriptions — include steps to reproduce, expected behavior, and actual behavior
- Review and respond to feedback on your pull requests promptly
- Close issues and pull requests that are no longer relevant

### Licensing Compliance

- Every project repository must include a valid FOSS license by the end of the event
- Do not include code, fonts, images, or other assets whose licenses conflict with your project's stated license
- When in doubt about licensing compatibility, consult the FOSS United resources or ask a mentor

---

## Mentorship and Learning

- Mentors are here to guide, not to build the project for you
- Approach mentors with specific, well-defined questions
- Be receptive to feedback and willing to change your approach when advised
- Share what you learn — open-source culture thrives on knowledge sharing

---

## Integrity and Fair Play

- All project contributions must be made by the registered team members
- Do not use AI-generated code as the primary basis for your project without disclosure and genuine understanding of the code
- Do not misrepresent the origin, timeline, or nature of your contributions
- Violations of integrity guidelines will result in disqualification

---

## Reporting Concerns

If you observe behavior that violates these guidelines or the Code of Conduct, report it to:

**fosshack@ggits.org**

All reports are treated with confidentiality and reviewed by the organizing committee.

---

## Changes to These Guidelines

The organizing committee reserves the right to update these guidelines at any time. Participants will be notified of significant changes through official event channels.
